# How BIP-322 works

BIP-322 proves control of a Bitcoin address over an arbitrary message by
constructing two throwaway ("virtual") transactions and signing the second
one as if it were spending the first. Nothing is ever broadcast.

## 1. The message hash

The message is hashed with a BIP-340 tagged hash, tag `BIP0322-signed-message`:

```
message_hash = SHA256(SHA256(tag) || SHA256(tag) || message)
```

`message` is the raw byte string — no length prefix, no null terminator, no
normalization. This is deliberately *not* the legacy
`"Bitcoin Signed Message:\n"` construction used by `signmessage`/BIP-137;
reusing that magic string is the single most common mistake when
implementing BIP-322 from scratch.

See `lib/src/crypto/tagged_hash.dart`.

## 2. The `to_spend` transaction

A transaction that "spends" a fictitious coin, challenging whoever holds the
address's key to prove it by "spending" this transaction's output:

```
nVersion    = 0
nLockTime   = 0
vin[0]:
  prevout   = (0x00...00, 0xFFFFFFFF)   // null outpoint
  scriptSig = OP_0 PUSH32[message_hash] // bytes: 00 20 <32-byte hash>
  nSequence = 0
vout[0]:
  nValue       = 0
  scriptPubKey = <the address being proven>
```

The scriptSig push must be minimal (`00 20 <hash>`, never a non-minimal
`4c 20 <hash>`) — a non-minimal push changes the transaction id, and every
downstream computation depends on it. See `lib/src/virtual_tx.dart` and
`lib/src/script.dart`.

## 3. The `to_sign` transaction

A transaction that spends `to_spend:0`, carrying the actual signature in its
witness (or scriptSig, for legacy inputs):

```
nVersion    = 0
nLockTime   = 0
vin[0]:
  prevout        = (to_spend.txid, 0)
  scriptSig      = empty (segwit) or the signature (legacy)
  nSequence      = 0
  scriptWitness  = the message signature (segwit)
vout[0]:
  nValue       = 0
  scriptPubKey = OP_RETURN
```

The signature hash committed here is computed exactly as if this were a real
transaction spending a real output of value 0 — this is the second common
mistake: forgetting that the segwit sighash (BIP-143 or BIP-341) commits to
the *spent output's amount*, which is always 0 here, not the sender's actual
balance. See `lib/src/sighash.dart` (legacy + BIP-143) and the sibling
[`bip341`](https://github.com/ethicnology/dart-bip341) package's
`lib/src/sighash.dart` (BIP-341).

## 4. Signing per address type

| Address | Sighash | Signature scheme | Witness/scriptSig |
|---|---|---|---|
| P2WPKH | BIP-143, scriptCode = implied P2PKH script, amount 0 | ECDSA (low-S, DER + `SIGHASH_ALL` byte) | `[sig, pubkey]` |
| P2TR (key-path) | BIP-341, `SIGHASH_DEFAULT`, amount 0 | BIP-340 Schnorr, 64 bytes | `[sig]` |
| P2WSH multisig | BIP-143, scriptCode = witnessScript | ECDSA per key | `[OP_0, sig..., witnessScript]` |
| P2PKH (legacy/BIP-137) | pre-segwit sighash | ECDSA, 65-byte recoverable | scriptSig (not a BIP-322 witness) |

Taproot key-path spends additionally require the **taptweak** (BIP-341):
the internal private key is tweaked with `t = tagged_hash("TapTweak", x_only(P))`
*before* signing — the output key committed to by the address is `Q`, not the
raw internal key. This is implemented in the sibling
[`bip341`](https://github.com/ethicnology/dart-bip341) package's
`lib/src/tweak.dart`, a dependency of `bip322` rather than embedded code
(only `elliptic`/`pointycastle`, independently tested against the official
BIP-341 vector — reusable outside BIP-322).

This implementation currently covers P2WPKH and P2TR (key-path); P2WSH
multisig is planned. **P2PKH and P2SH-P2WPKH are deliberately out of
scope**: `bip322`'s `Bip322.signMessage`/`Bip322.verify` throw a Dart
`UnsupportedError` for both, even though `parseAddress` recognises them —
P2PKH's scheme (BIP-137) isn't BIP-322 proper, and P2SH-P2WPKH needs the
Full format's scriptSig slot, which Simple doesn't have (see §5). See
`Bip322._rejectOutOfScopeAddressType` in `lib/src/bip322.dart`.

## 5. The "Simple" signature format

The most common format ("Simple", prefix `smp`) is just the witness stack of
`to_sign`'s single input, consensus-encoded and base64'd:

```
compactSize(item_count) || (compactSize(len) || bytes)*
```

This is *not* a full transaction — that's the "Full" format (`ful`), a
distinct, larger encoding reserved for cases the witness alone can't express
(e.g. P2SH-P2WPKH, which needs a scriptSig slot that Simple doesn't have).
See `lib/src/witness.dart` and `lib/src/formats.dart`.

## 6. Verifying

To verify: reconstruct `to_spend` and `to_sign` from the claimed
`(message, address)`, recompute the sighash for the address's type, and
check the signature in the provided witness against the address's key
material (recovered from the witness itself for P2WPKH/P2WSH, or read from
the address for P2TR).

A correct verifier must bind the *decoded signature format* to the *address
type* — a legacy 65-byte blob must never be accepted for a P2TR address, and
vice versa — otherwise a signature crafted for one context could be replayed
as if valid in another. See `lib/src/verifier.dart` and the "type confusion"
tests in `test/m6_hardening_test.dart`.
